# The most direct changes I would make to your updated plan

These are the highest-value changes, not generic polish.

## 1. Do **not** make live settings modify repo files

This is the first thing I would change.

Right now the plan still implies things like:

* background color → `static/index.html`
* mission focus → `core/mission_profile.json`
* speed profile → `core/runtime_prefs.json`

That is too close to source-code mutation for routine user preferences.

I would split this into three layers:

* **repo defaults**: checked into git
* **persistent user overrides**: DB or app-support config store
* **session overlays**: in-memory only

So instead of editing source files for routine changes:

* theme color updates `runtime_overrides.theme_tokens`
* mission focus updates `mission_overrides`
* speed changes update `runtime_overrides.response_profile`

Then the browser and reflex layer read from the override store, not from edited source.

This matters because otherwise:

* normal user preferences dirty the repo
* Forge diffs get polluted with preference noise
* rollbacks get messy
* blue/green candidates can disagree about “the code” vs “the current user state”

This is a **must-change**.

## 2. Split `reply debt` from `turn activity`

Right now `pending_user_turn_id` is doing too much.

I would split it into:

* `reply_debt_id`: user is still owed an answer
* `active_turn_id`: this turn is still operationally alive
* `floor_owner`: user / assistant / none

Why this matters:

* once JARVIS starts substantively answering, `reply_debt` can clear
* but the turn may still be active until playback completion
* background cognition should still stay blocked until turn close
* this makes logs, testing, and replay much cleaner

Without this split, you risk weird edge cases where a turn is no longer “unanswered” but the system still conflates that with cooldown / engaged silence.

## 3. Add an explicit **Validator** lane and **Renderer** lane

Your functional roles are strong, but I would add two more explicit roles:

* **Validator**: judges evidence, confidence, and memory eligibility
* **Renderer**: turns structured private cognition into user-facing summaries

Right now the plan has anti-psychologizing validation, which is good, but I’d make validation a first-class lane, not just a guardrail.

Why:

* Dreamer without Validator will still generate beautiful junk
* Researcher/Mechanic outputs should not go straight into memory or initiative
* user-visible internal dialogue should be rendered from validated frames, not raw `inner_voice`

This is one of the biggest upgrades for making the inner life feel **real** instead of just well-structured.

## 4. Add a real persistent **Quest table**, not just `quest_id` fields

The plan introduces `quest_id`, which is a strong move, but I would make quests a first-class DB object.

Add something like:

* `quests(id, kind, title, mission_axis, status, salience, created_at, last_progress_at, evidence_refs, owner_lane, surfaceability, closure_reason)`

Why:

* interesting inner life needs **long-lived threads**
* the user should be able to ask “what are your active quests?”
* repairs, science questions, and self-improvement efforts need continuity
* reward can now attach to quest progress, not just isolated pulses

This is the missing persistence layer for “digital dopamine around discovery.”

## 5. Compile `SELF_PACK` primarily from **runtime manifests and contracts**, not directly from the Bibles

This is a major concern.

Your docs contain a mix of:

* current implementation
* target design
* planned phases
* aspirational behavior

If `SELF_PACK` is built from “Bibles + runtime,” JARVIS can still overclaim.

I would make the truth hierarchy:

1. **runtime state**
2. **module contracts / capability registry / edit policies**
3. **versioned manifests**
4. **docs as human-readable annotations only**

And every self-model field should carry a status:

* `implemented`
* `experimental`
* `planned`
* `disabled`

This is critical if you want JARVIS to answer questions about itself truthfully.

Otherwise it will still say things that are true of the *architecture doc* but not the *running system*.

## 6. Add `presence_conf` and a social budget back into initiative

The current updated plan is great on floor sovereignty, but initiative still needs a second gating layer:

* is the floor free?
* **and is this a good moment socially?**

I would add:

* `presence_conf`
* `initiative_budget`
* `recent_initiative_acceptance_rate`

The floor being free is necessary, but not sufficient.

This prevents JARVIS from surfacing good ideas at bad moments.

## 7. Do not silently drop initiative candidates

Your initiative worker currently says “silently drop” if the floor is no longer idle.

I would change that.

Better behavior:

* if the candidate is stale or low salience, shelve it
* if it is still relevant, requeue with TTL
* always log suppression reason

This matters because otherwise the system loses useful thoughts and becomes harder to debug.

## 8. Replace the hard 30-second clamp with **segmented finalize**, plus dictation mode

The 302-second ASR accumulation bug is real and severe, but a hard 30-second ceiling will break legitimate long questions.

Better design:

* at 30 seconds, force-finalize the current segment
* continue capture into a new segment if speech continues
* preserve a `turn_group_id`
* add explicit dictation mode if needed later

That gives you protection without punishing long-form speech.

## 9. Add protocol/version compatibility as a first-class contract

This is the biggest hidden break risk in the whole plan.

If Forge edits:

* WebSocket events
* browser message handling
* audio protocol behavior
* UI JavaScript expectations

then blue/green swap can fail because the old browser page is still live while the new reflex slot expects a new protocol.

I would add:

* `ws_protocol_version`
* `ui_asset_version`
* `compatible_with_versions`
* `requires_full_reload`

And test it.

This needs to live in:

* `module_contracts.yaml`
* candidate verifier
* routerd swap logic

Otherwise the first serious UI/protocol self-edit can brick the live session.

This is a **major concern**.

## 10. Candidate verification must not share mutable live side effects by default

Another major concern.

If candidate B boots against the same live DB/event streams as A during replay or smoke tests, you can get:

* double consumers
* duplicate writes
* weird initiative duplication
* racing spool/event handling
* corrupted eval metrics

So I would make candidate environments use one of:

* shadow DB snapshot
* read-only replay DB
* side-effect fence
* shadow queues

Only after promotion should the candidate take the live mutating role.

This is especially important for:

* spool/event writer
* thoughts / initiative persistence
* eval runs
* question / experiment updates

## 11. Add chaos tests, not just replay tests

Replay tests are great, but your most dangerous failures are racey.

I would add fault injection for:

* dropped `tts_finished`
* websocket disconnect mid-utterance
* barge-in during candidate swap
* delayed `interrupt_hint`
* stale audio chunk after epoch bump
* mind pulse cancelled mid-JSON
* reconnect with old UI bundle to new server slot

Your bug came from coordination failure, not from a pure logic branch. Chaos tests matter here.

## 12. Add a sanctioned user-visible “Mind Inspector v2”

This is important for the original complaint.

The updated plan fixes the cognition substrate, but the user still needs a truthful way to **see** the internal dialogue.

I would add disclosure modes:

* `off`
* `summary`
* `quests`
* `live_rendered`
* `raw_debug` (developer-only)

And the UI should show things like:

* active quests
* recent repairs
* recent experiments
* recent validated thought summaries
* what was suppressed and why

Not raw scratchpads. Not secret reasoning leakage.
But not a blank box either.

This is how the system stops feeling like it has “no internal dialogue.”

# Major concerns in the updated plan

These are the places most likely to break badly if left as-is.

## Major concern 1: protocol drift between browser and swapped control plane

This is my biggest concern.
A/B swapping reflex while allowing self-edits to `static/index.html` / protocol behavior is dangerous without explicit compatibility/version rules.

## Major concern 2: live settings are still too repo-like

User preferences should not be source-code edits or git-tracked JSON mutations.

## Major concern 3: shared-state candidate boot

If A and B both touch live mutable state during verification, you will get ghosts, duplicates, and very hard-to-debug regressions.

## Major concern 4: JSON-only mind pulses may fail more than expected

The direction is correct, but local model reliability matters. I would strongly prefer schema-constrained generation or a deterministic local repair/parser before final discard.

## Major concern 5: the scheduler can still ruminate if fairness is too simple

A max-score Agenda Broker can get stuck chasing the same repair or curiosity thread. Add:

* cooldowns
* aging
* diversity bonus
* quest anti-starvation
* per-lane quotas

## Major concern 6: the time estimate is optimistic

The plan is excellent, but **20–28 days** is aggressive for:

* monolith split
* new daemons
* routerd
* code atlas
* forge
* replay + chaos harness
* version contracts

The design is strong. The schedule is the shakier part.

# What I would add that is not in the plan yet

These are genuinely new upgrades I think would make it even better.

## A. `turn_group_id`

Use one ID to group segmented long speech, barge-ins, retries, partial ASR, and final answer artifacts. That will make replay and debugging dramatically easier.

## B. `capability escalation guard`

Any self-edit that increases:

* network access
* file scope
* shell scope
* external tool access
  should require elevated approval, even if the files are technically mutable.

## C. `kept-change reward`

A self-edit should not only be rewarded for passing tests. It should be rewarded more if:

* the user keeps it
* it survives 24 hours
* it reduces regressions
* it improves latency or satisfaction

## D. `safe switch point`

Even with blue/green, do not swap mid-turn. Switch only when:

* floor is `TRULY_IDLE` or `MAINTENANCE`
* audio queue is drained
* no pending reply debt
* no active ASR
* browser has acknowledged maintenance begin

## E. `docs-runtime drift audit`

Run a periodic job that compares:

* self-manifest
* module contracts
* capability registry
* runtime state
  against the human bibles, and flags mismatches.

That keeps self-knowledge honest as the system evolves.
